manifest.json — the rulebook, decoded

Every Chrome extension starts with a manifest. It's a JSON file Chrome reads before loading anything else.

CODE 

{
  "manifest_version": 3,
  "name": "Tab Out",
  "permissions": ["tabs", "activeTab", "storage"],
  "chrome_url_overrides": {
    "newtab": "index.html"
  },
  "background": {
    "service_worker": "background.js"
  }
}
PLAIN ENGLISH

Use the latest version of Chrome extension rules (version 3).

This extension is called "Tab Out".

We need 3 permissions: read all tabs, access the active tab, and save data locally.

Override Chrome's built-in new tab page...

...with our own index.html file.

Also run a background script...

...called background.js, which runs as a service worker (always-on, no visible UI).

🔑
Permissions = what the extension is allowed to do

When users install Tab Out, Chrome shows a popup asking them to approve these permissions. The extension can only do what's listed here — it can't quietly read your passwords or camera. If you ever want to add a new capability to Tab Out (say, sync tabs across devices), you'd need to add a new permission here first.

tabs Read the URL, title, and window of every open tab
activeTab Access the tab the user is currently looking at
storage Save data to the browser between sessions (the "Saved for Later" list)

How Tabs Get Organized

The sorting algorithm that turns a chaotic list of URLs into clean, grouped domain cards.

Chrome gives you a flat list. Tab Out makes it useful.

When app.js asks Chrome for your open tabs, it gets back a flat pile of objects — every tab in every window, in no particular order. Imagine dumping 40 tabs on the floor.

The grouping logic in app.js is like a very fast librarian: it sorts everything by domain in milliseconds, with two special rules that make Tab Out genuinely useful.

🏠

Landing Pages get their own group

Gmail inbox, Twitter home, YouTube home — these "homepage" tabs are pulled out into their own "Homepages" card, separate from your actual work tabs on those same sites.

🔗

Custom groups let you merge domains

You can configure rules to group related domains together — like merging all Notion subdomains into one card. Defined in a personal config file that isn't shared.

The heart of the grouping logic

This is the loop that sorts every open tab into its bucket. It runs inside renderStaticDashboard() in app.js.

CODE 

for (const tab of realTabs) {
  try {
    if (isLandingPage(tab.url)) {
      landingTabs.push(tab);
      continue;
    }
    const hostname = new URL(tab.url).hostname;
    if (!groupMap[hostname]) {
      groupMap[hostname] = { domain: hostname, tabs: [] };
    }
    groupMap[hostname].tabs.push(tab);
  } catch { /* skip malformed URLs */ }
}
PLAIN ENGLISH

Go through every real tab one by one.

Try to process it (some URLs are weird and will crash — catch those).

Is this a "homepage" URL (Gmail inbox, YouTube home, etc.)?

If yes, add it to the special landing-pages pile...

...and skip to the next tab.

Otherwise, extract the domain from the URL (e.g. "github.com" from a long URL).

If we haven't seen this domain before, create a new bucket for it.

Either way, add this tab to that domain's bucket.

If the URL is broken, quietly skip it.

💡
This pattern is called "bucketing"

Sorting items into named buckets is one of the most common patterns in programming. You've seen it in analytics (sessions grouped by country), in finance (transactions grouped by category), and in email clients (messages grouped by sender). Whenever you need to transform a flat list into groups, reach for this pattern.

How it knows "this is a homepage"

The isLandingPage() function checks each URL against a list of known homepage patterns. It has to be specific — closing "Gmail inbox" shouldn't also close threads you have open in Gmail.

CODE 

const LANDING_PAGE_PATTERNS = [
  { hostname: 'mail.google.com', test: (p, h) =>
      !h.includes('#inbox/') && !h.includes('#sent/') },
  { hostname: 'x.com', pathExact: ['/home'] },
  { hostname: 'github.com', pathExact: ['/'] },
];
PLAIN ENGLISH

A list of "these are homepage URLs" rules.

Rule 1: Gmail — it IS a homepage, UNLESS the URL contains an email thread ID (those are content, not the inbox).

Rule 2: X.com — only the /home path is the homepage. Profiles and posts are not.

Rule 3: GitHub — only the root "/" is the homepage. Repos and issues are content.

Click, Close, Confetti

How one listener handles every button in the app — and why the swoosh sounds so satisfying.

One guard for the whole building

Tab Out has dozens of buttons — close group, close single tab, save for later, close duplicates, expand overflow. Naively, you'd put a event listener on every single button. Tab Out doesn't — it uses one listener on the entire document.

This pattern is called event delegation. Think of it like building security: instead of a guard at every door, there's one guard in the lobby who watches everyone who walks in and checks their badge.

💡
Event delegation — why it matters for you

Since Tab Out generates card HTML dynamically (new buttons appear whenever tabs are read), you can't attach listeners to buttons that don't exist yet. Event delegation solves this: the listener is on the document, which always exists. When you describe interactive UIs to AI, say "use event delegation so listeners survive dynamic content re-renders."

The single click handler — decoded

One async function handles every button click in Tab Out. Here is how it decides what to do.

CODE 

document.addEventListener('click', async (e) => {
  const actionEl = e.target.closest('[data-action]');
  if (!actionEl) return;
  const action = actionEl.dataset.action;
  if (action === 'close-domain-tabs') {
    /* ... close all tabs in group */
  }
  if (action === 'close-single-tab') {
    /* ... close just one tab */
  }
});
PLAIN ENGLISH

Listen for ANY click anywhere in the document.

Walk up the DOM from where the user clicked, looking for an element with a "data-action" label.

If we found nothing with a data-action label, this click isn't meant for us — ignore it.

Read the action name from the element's data attribute.

If the action is "close all tabs in a domain group"...

...run the close-group logic.

If the action is "close just this one tab"...

...run the single-tab-close logic.

How the swoosh sound is made — from scratch

Tab Out plays a sound when you close tabs, but there is no MP3 file anywhere in the codebase. The sound is generated mathematically using the Web Audio API — every time, in real time.

Think of it like the difference between playing a recording of a piano versus simulating the physics of a piano string. Tab Out does the latter.

CODE 

const ctx = new AudioContext();
const buffer = ctx.createBuffer(1, ctx.sampleRate * 0.25, ctx.sampleRate);
const data = buffer.getChannelData(0);
for (let i = 0; i < data.length; i++) {
  const pos = i / data.length;
  const env = pos < 0.1 ? pos / 0.1 : Math.pow(1 - (pos - 0.1) / 0.9, 1.5);
  data[i] = (Math.random() * 2 - 1) * env;
}
const filter = ctx.createBiquadFilter();
filter.frequency.setValueAtTime(4000, t);
filter.frequency.exponentialRampToValueAtTime(400, t + 0.25);
PLAIN ENGLISH

Open an audio processing context (like turning on the synthesizer).

Create 0.25 seconds of blank audio memory — a canvas for our sound.

Get access to the raw audio samples in that memory.

Loop through every single audio sample (there are thousands).

Calculate how far through the sound we are (0 = start, 1 = end).

Build an envelope shape: ramp UP fast in the first 10%, then fade out smoothly. This gives the "whoosh" feel.

Fill each sample with random noise, shaped by that envelope.

Create a frequency filter (like a tone knob on a stereo).

Start the filter at 4000Hz (high, bright, airy)...

...and sweep it down to 400Hz over 0.25 seconds. This is the "swoosh" — high to low.

🎵
Sound as math — a transferable superpower

The Web Audio API is available in every browser. Any web app you build — not just extensions — can generate custom sounds without audio files. Notification sounds, game audio, UI feedback — all doable in pure JavaScript. Next time you need a sound effect, ask your AI: "generate this using the Web Audio API, no audio files."

Your Tab Memory

How Tab Out remembers your saved tabs across sessions — without any server or database.

A library card catalog, built into your browser

When you click the bookmark icon on a tab, Tab Out needs to remember it — even after you close the browser and come back tomorrow. Normally this would require a server and a database.

Tab Out uses chrome.storage.local instead — a tiny key-value store that Chrome manages for each extension. No servers, no accounts, no monthly bills.

💾
Persists across sessions

Data survives browser restarts. Your saved tabs are still there Monday morning.

🔒
Private by design

Stored locally on your machine. Tab Out has no idea what you saved — no data ever leaves Chrome.

⚡
Instant reads and writes

No network round-trip. Saving a tab is as fast as writing to memory.

What gets stored — and how

Everything lives under a single key called "deferred". The value is an array of saved tab objects, each with a handful of fields.

CODE 

async function saveTabForLater(tab) {
  const { deferred = [] } =
    await chrome.storage.local.get('deferred');
  deferred.push({
    id:        Date.now().toString(),
    url:       tab.url,
    title:     tab.title,
    savedAt:   new Date().toISOString(),
    completed: false,
    dismissed: false,
  });
  await chrome.storage.local.set({ deferred });
}
PLAIN ENGLISH

Define a function that saves one tab for later.

Read the current saved-tabs list from storage. If nothing is stored yet, start with an empty list.

Add a new item to the list, with these fields:

A unique ID — just the current timestamp (milliseconds since 1970). Good enough as a unique number.

The tab's URL — where it points.

The tab's title — what to show in the checklist.

When it was saved — stored as an ISO date string (e.g. "2026-04-16T10:00:00Z").

completed: false — not checked off yet.

dismissed: false — not removed yet.

Write the updated list back to storage. Done — it will survive a browser restart.

Three states, two flags

Every saved tab moves through a simple lifecycle, controlled by just two boolean flags: completed and dismissed.

1
Active — completed: false, dismissed: false

The tab appears in the checklist. The user can read it, check it off, or dismiss it.

2
Archived — completed: true, dismissed: false

The user checked off the tab. It moves to the collapsible archive section. Still retrievable.

3
Gone — dismissed: true

The user clicked X. The tab is hidden from all views. Still in storage, but filtered out everywhere.

💡
Soft delete — never destroy data you might need

Notice that Tab Out never actually removes items from the storage array — it just sets a flag. This is called a "soft delete" pattern. The benefit: you can add an "Undo" feature later, or an analytics view, without needing to reconstruct lost data. When you describe deletion features to AI, ask for soft delete unless you truly need hard delete.
